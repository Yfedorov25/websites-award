# D_ActiveTheory — activetheory.net (ГЛИБИННИЙ РОЗБІР, 100% за схемою v1) ✅
> СВІТОВИЙ ЕТАЛОН. Багаторазовий Awwwards Site of the Year. LA/Amsterdam. Власний движок Hydra.
> Розбір desktop 1280×660 live. Найрадикальніший «scene-as-site» — майже все у WebGL.

## 1. ТЕХ-СТЕК
- **vanilla + WebGL2, 1 canvas, 3 скрипти, 8 Web Workers** (`hydra-thread.js` ×8 — multi-threaded движок Hydra).
- THREE/GSAP/Lenis НЕ на window — повністю власний движок.
- **АРХІТЕКТУРА:** docHeight=660=1vp (body не скролиться, усе в canvas). Скрол перехоплений движком.
- **Контент-стратегія:** DOM-текст лише 111 символів (тільки UI-хром: Toggle Audio/Work/Contact/музплеєр). 0 заголовків у DOM, 107 елементів усього. УВЕСЬ контент-текст = геометрія/текстури у WebGL. Найрадикальніший scene-as-site (радикальніше за Lusion-гібрид).

## 2. ДИЗАЙН-ТОКЕНИ
- **CSS майже нема** (1 :root змінна --baropacity). Бо вся візуальна система — у шейдерах/движку, не CSS.
- Шрифт: **nbarchitekt** (один, монопросторовий технічний). Чорна база #000 (і bg, і color — текст у canvas).
- Урок: коли все у WebGL, дизайн-токени живуть у движку (JS/шейдери), не в CSS. CSS лише для UI-хрому.

## 3. DOM-АРХІТЕКТУРА Й ШАРИ
- 1 canvas (повноекранний, повний рендер). DOM-оверлей мінімальний: аудіо-тогл, нав (Work/Contact), музплеєр (Song--Artist, <</>>).
- Усе інше (заголовки, проєкти, переходи) — у canvas через Hydra.

## 4. ПОЕКРАННА РОЗКАДРОВКА
- Контент динамічний з CMS (нижче). Заголовки/проєкти рендеряться у WebGL — для дослівного копірайту потрібен візуальний знімок (DOM порожній). Нав: Work, Contact. Фішка: вбудований музичний плеєр (Toggle Audio) — сайт як аудіовізуальний досвід.

## 5. ПЕРЕХОДИ / РУХ
- Усі переходи — у WebGL (Hydra рендерить сцени й переходи між ними). Multi-threaded = плавність навіть на важких сценах.

## 6. КОПІРАЙТ / СЕНСИ
- Title/позиція: «Active Theory · Creative Digital Experiences». Продають «досвіди», не сайти. Аудіо-перший підхід (музика як частина бренду).

## 7-8. ЕМОЦІЙНА ПАРТИТУРА
- Повне занурення: чорний екран + WebGL-сцена + музика = «ти всередині досвіду, не на сайті». Аудіо підсилює іммерсію (рідкісний хід).

## 9. ВІДТВОРЮВАНІСТЬ (як це зроблено й чи повторимо) ★★★
### Движок Hydra — інженерна анатомія (з network):
- **8× hydra-thread.js** = 8 Web Workers паралельно. Рендер/фізика/завантаження рознесені по потоках → плавність на важких сценах. ЯК: Worker pool, OffscreenCanvas, передача даних через postMessage/SharedArrayBuffer.
- **compiled.vs** = прекомпільований шейдер (збирають шейдери білд-часом, не рантайм) → швидший старт.
- **Draco .wasm** = стиснення 3D-геометрії (Google Draco) → mesh у рази менші.
- **Basis transcoder .wasm** = GPU-стиснені текстури (Basis Universal) → менший трафік, декод на GPU.
- **Headless CMS на Google Cloud Storage** (cms JSON) = статичний движок + динамічний контент-JSON. Контент-команда міняє JSON, не чіпає движок.

### Чи повторюється:
- **🔴 НЕДОСЯЖНО 1:1** — власний multi-threaded движок Hydra = роки R&D команди. Не копіюємо.
- **🟡 ЩО ПЕРЕЙМАЄМО (досяжні принципи):**
  - Draco-стиснення геометрії + Basis/KTX2-текстури → у R3F через `useGLTF` з Draco + `KTX2Loader`. Дає ту саму оптимізацію БЕЗ власного движка. 🟢-🟡 (drei це вміє).
  - Headless CMS (контент окремо від коду) → Sanity/Contentful + JSON. 🟢.
  - OffscreenCanvas + Worker для важких обчислень → доступно в R3F. 🟡.
  - Прекомпіляція шейдерів → vite-plugin-glsl. 🟢.
- **🟢 АЛЬТЕРНАТИВА для «scene-as-site» враження:** замість власного движка — R3F + drei + postprocessing з 1-2 фокусними 3D-сценами (не весь сайт). 80% враження за 20% зусиль.

### АЛЬТЕРНАТИВА якщо клієнту треба «як Active Theory», а бюджету нема:
- Pre-rendered відео-фон ключової сцени (Higgsfield/Blender) замість real-time движка. Виглядає так само, коштує в рази менше. Real-time лишати тільки там, де потрібна інтерактивність.
